# 1. 简介

建造者模式（Builder）是一种**创建型设计模式**，**旨在将复杂对象的构建过程与其表示分离，使得同样的构建过程可以创建不同的表示形式**。通过使用建造者模式，可以一步一步地创建一个复杂的对象，并且可以根据需要改变对象的内部表示。

**主要目标是将对象的构建过程分离出来，使得相同的构建过程可以创建出不同的对象**。这种设计模式特别**适用于需要逐步创建复杂对象的场景**，因为它允许客户端在不知晓对象构建细节的情况下创建对象。

# 2. 核心思想

• **分步构建**：通过分步构建的方式，可以更好地控制对象的创建过程，并根据需要灵活调整每一步的内容。
• **分离复杂对象的构建和表示**：将对象的构建过程与其表示分离，使得构建代码和对象代码可以独立变化。

# 3. 主要组成部分

1. **产品类（Product）**：表示要创建的复杂对象。产品类包含了多个组件，每个组件通常由不同的构建步骤来创建。
2. **抽象建造者（Builder）**：定义了构建产品的步骤接口，包含创建产品组件的抽象方法。这些方法由具体建造者实现。
3. **具体建造者（ConcreteBuilder）**：实现 Builder 接口的具体类，负责构建产品的各个组件，并最终组装成产品对象。
4. **指挥者（Director）**：负责调用建造者的构建步骤来完成产品的构建。指挥者并不依赖于具体建造者，只依赖于 Builder 接口，因此可以通过不同的建造者创建不同的产品。
5. **客户端代码（Client）**：通过指挥者和建造者来获取最终的产品对象。客户端不直接与产品类交互，而是通过建造者来间接构建产品。

# 4. 适应场景

建造者模式适用于以下场景：

1. **需要一步步构建复杂对象**：当创建一个复杂对象需要多个步骤时，建造者模式可以帮助将构建过程分解为多个可控的步骤，从而使得构建过程更加清晰和灵活。
2. **需要创建不同形式的对象**：当需要通过同样的构建过程创建不同形式的对象时，建造者模式可以通过替换建造者来实现这一需求。
3. **需要隔离复杂对象的构建与表示**：当需要将对象的构建过程与其表示分离时，建造者模式可以将这些职责分开，从而使得代码更加易于理解和维护。

**举例**

例如，一个房屋建造系统可能需要支持不同风格的房屋建造。通过建造者模式，可以定义一个房屋建造的通用流程（如建造地基、搭建结构、安装屋顶等），然后通过不同的建造者来创建不同风格的房屋（如现代风格、古典风格）。这样，无论是修改建造过程还是新增房屋风格，都不需要对已有代码进行大量修改。

# 5. 优缺点

## 5.1 优点

1. **简化创建过程**：建造者模式将复杂对象的创建过程分解为一系列可控的步骤，使得创建过程更加清晰、易于维护。
2. **代码复用性高**：不同的建造者可以复用相同的构建步骤，从而提高了代码的复用性。
3. **易于扩展**：通过引入新的建造者类，可以轻松扩展系统，创建新的产品表示，而无需修改已有的构建过程。
4. **遵循单一职责原则**：建造者模式将对象的创建与表示分离，构建过程由专门的建造者类负责，使得代码职责更加清晰。

## 5.2 缺点

1. **可能引入额外的复杂性**：当产品的构建步骤过多时，建造者模式可能会引入大量的建造者类，从而增加代码的复杂性。
2. **适用性有限**：建造者模式适用于需要创建复杂对象的情况，但对于简单对象的创建，可能显得过于笨重，不够灵活。

## 6.3 代码解析

这是一个基于建造者模式的 Java 代码结构。以下是对每个类的解释：

### 6.3.1 **Client.java**

- **位置**: `com.wangguangwu.Client`
- **角色**: 客户端代码。
- **说明**: 这是建造者模式的客户端类，负责初始化 `Director` 和 `Builder` 对象，并通过 `Director` 来构建产品对象。客户端调用 `Director` 的方法来构建复杂的产品对象，而无需直接操纵 `Builder` 的具体实现。

### 6.3.2 **builder 包**

这个包包含了建造者模式中的 `Builder` 接口及其具体实现类。

#### 6.3.2.1 **Builder.java**

- **位置**: `com.wangguangwu.builder.Builder`
- **角色**: 抽象建造者接口。
- **说明**: 定义了创建复杂对象的各个部分的抽象方法，如 `reset()`、`buildStepA()`、`buildStepB()` 和 `getResult()`。具体的 `Builder` 实现类需要提供这些方法的具体实现。

#### 6.3.2.2 **impl 包**

这个子包包含了 `Builder` 接口的具体实现类。

- **ConcreteBuilder1.java**

    - **位置**: `com.wangguangwu.builder.impl.ConcreteBuilder1`
    - **角色**: 具体建造者 1。
    - **说明**: 实现了 `Builder` 接口，提供了构建 `Product1` 对象的具体步骤。每个构建步骤都会调用相应的方法，以逐步构建 `Product1` 对象。最终，通过 `getResult()` 方法返回构建完成的 `Product1` 对象。

- **ConcreteBuilder2.java**
    - **位置**: `com.wangguangwu.builder.impl.ConcreteBuilder2`
    - **角色**: 具体建造者 2。
    - **说明**: 实现了 `Builder` 接口，提供了构建 `Product2` 对象的具体步骤。与 `ConcreteBuilder1` 类似，`ConcreteBuilder2` 负责创建 `Product2` 对象。

### 6.3.3 **director 包**

这个包包含了 `Director` 类。

- **Director.java**
    - **位置**: `com.wangguangwu.director.Director`
    - **角色**: 指挥者类。
    - **说明**: `Director` 类负责定义构建产品对象的步骤顺序。它会调用 `Builder` 提供的构建步骤方法，并在合适的时机调用 `Builder` 的 `getResult()` 方法来获取最终的产品对象。`Director` 可以使用不同的 `Builder` 来创建不同类型的产品。

### 6.3.4 **product 包**

这个包包含了产品类及其具体实现。

#### 6.3.4.1 **Car.java**

- **位置**: `com.wangguangwu.product.Car`
- **角色**: 通用产品类。
- **说明**: `Car` 类代表了一种可以由 `Builder` 构建的通用产品。它可以作为其他具体产品类（如 `Product1` 和 `Product2`）的基类，或者单独用作产品。

#### 6.3.4.2 **Product1.java**

- **位置**: `com.wangguangwu.product.Product1`
- **角色**: 具体产品 1。
- **说明**: `Product1` 类代表由 `ConcreteBuilder1` 构建的具体产品。它包含了产品的具体特性和行为。

#### 6.3.4.3 **Product2.java**

- **位置**: `com.wangguangwu.product.Product2`
- **角色**: 具体产品 2。
- **说明**: `Product2` 类代表由 `ConcreteBuilder2` 构建的具体产品。它包含了产品的具体特性和行为。

### **总结：**

- **`Client.java`**: 客户端代码，使用 `Director` 和 `Builder` 来创建和获取复杂产品对象。
- **`Builder.java`**: 抽象建造者接口，定义了创建产品各部分的步骤。
- **`ConcreteBuilder1.java` 和 `ConcreteBuilder2.java`**: 具体建造者类，实现了 `Builder` 接口，负责构建具体的产品对象。
- **`Director.java`**: 指挥者类，负责定义构建产品的顺序。
- **`Car.java`, `Product1.java`, `Product2.java`**: 产品类，代表了由 `Builder` 构建的具体产品。

通过这种结构，建造者模式将产品的构建过程与其具体表示分离开来，使得代码更灵活，易于扩展。

# **7. 建造者模式和工厂模式的异同**

### 7.1 相同点

1. **创建型设计模式**：建造者模式和工厂模式都是创建型设计模式，它们的主要作用都是帮助客户端创建对象，并隐藏对象创建的复杂性。

2. **解耦对象创建和使用**：两者都将对象的创建与使用解耦。客户端不需要知道对象的创建细节，只需通过接口或抽象类获取对象即可。这种解耦使得系统更具灵活性和可扩展性。

3. **遵循开闭原则**：两者都遵循开闭原则，通过引入新的具体实现（如新的工厂类或建造者类）来扩展系统功能，而无需修改现有的代码结构。

4. **支持不同类型的对象**：无论是工厂模式还是建造者模式，都可以通过不同的具体实现类来创建不同类型的对象。这使得系统能够灵活地应对变化和扩展。

### 7.2 不同点

1. **关注点不同**：

    - **工厂模式**：工厂模式主要关注“**对象的创建**”。它定义了一个接口或抽象类，通过子类来决定要实例化的对象类型。工厂模式的核心在于通过工厂方法创建对象，通常用于创建单个对象或一组相关对象。
    - **建造者模式**：建造者模式则关注“**对象的构建过程**”。它将对象的创建过程拆分为多个步骤，通过这些步骤的组合来构建复杂的对象。建造者模式特别适用于创建结构复杂、步骤多样的对象。

2. **复杂性**：

    - **工厂模式**：工厂模式通常用于创建简单或相对简单的对象。虽然也可以创建复杂对象，但工厂模式主要负责对象的创建逻辑，而不涉及对象构建的复杂过程。
    - **建造者模式**：建造者模式适用于创建复杂对象，尤其是那些需要分步骤构建的对象。通过将对象的构建过程封装在不同的步骤中，建造者模式使得构建过程更加可控且易于理解。

3. **产品结构的适用性**：

    - **工厂模式**：适用于创建一个产品或一组相关产品。它通常会根据传入的参数或环境条件返回不同的产品实例。
    - **建造者模式**：适用于构建具有复杂内部结构的产品。通过不同的建造者，可以构建出包含多个部分或属性的复杂对象。

4. **使用场景**：

    - **工厂模式**：当您需要根据不同的条件创建不同类型的对象时，工厂模式是理想的选择。它适合用于系统中需要灵活替换或扩展对象类型的场景。
    - **建造者模式**：当对象的创建步骤比较复杂且步骤之间存在依赖时，建造者模式是更好的选择。它适合用于系统中需要创建复杂对象、需要构建步骤灵活可控的场景。

5. **典型结构**：
    - **工厂模式**：典型的工厂模式包括一个抽象工厂类或接口和多个具体工厂类。每个具体工厂类负责创建一种具体产品。
    - **建造者模式**：典型的建造者模式包括一个抽象建造者接口或类、多个具体建造者类、一个产品类，以及一个指挥者类。指挥者类负责控制构建过程的顺序。

### 7.3 总结

- **工厂模式**注重对象的“**创建**”，适用于简单对象或一组相关对象的实例化，通过工厂方法将创建逻辑封装起来，使得客户端可以灵活地获取不同类型的对象。

- **建造者模式**注重对象的“**构建过程**”，适用于复杂对象的创建。它将构建过程分解为多个步骤，并通过指挥者类来控制构建顺序，使得客户端可以更灵活地构建复杂对象。

在实际开发中，选择哪种模式取决于系统的需求。如果需要简单的对象创建且类型可以根据条件变化，工厂模式是更好的选择；而如果需要构建复杂对象且构建过程可控，建造者模式则更为合适。

# 8. 总结

建造者模式是一种非常实用的设计模式，特别适用于构建复杂对象的场景。它通过分离构建过程和对象表示，使得代码更具可读性、可扩展性，并且易于维护。虽然它可能会增加类的数量和代码的复杂性，但其带来的优势往往能够抵消这些缺点，特别是在需要构建复杂对象的系统中。
